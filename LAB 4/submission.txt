AIM

To generate a lexical analyzer (scanner) and a syntax analyzer (parser) using the tools LEX and YACC and to study their behavior on declaration statements.

---

P1 
In this version:

- "i" represents int
- "f" represents float
- "a, b" represent identifiers

Example:

i a , b ;

represents:

int a,b;

Commands Used

lex declv1.l
yacc -d declv1.y
gcc lex.yy.c y.tab.c -o decl1
./decl1 < v1inp1.txt > v1out1.txt
./decl1 < v1inp2.txt > v1out2.txt
./decl1 < v1inp3.txt > v1out3.txt

P2 
In this version actual C-like declarations are used:

int a,b;

Commands

lex declv2.l
yacc -d declv2.y
gcc lex.yy.c y.tab.c -o decl2
cat y.tab.h
./decl2 < v2inp1.txt > v2out1.txt
./decl2 < v2inp2.txt > v2out2.txt
./decl2 < v2inp3.txt > v2out3.txt

Observation of y.tab.h

The file "y.tab.h" contains token definitions such as:

#define INT 257
#define ID 258

Explanation:
This file is included in the lex program so that both scanner and parser use the same token numbers.

---

Observations (P2)
Rule 2 used : SEEN INT
Rule 4 used :seen First Identifier
Rule 3 used: Seen another Identifier
Rule 3 used: Seen another Identifier
Rule 1 used : Successful Parse



Rule 2 used : SEEN INT
Rule 4 used :seen First Identifier
 syntax error  near symbol ,


Rule 2 used : SEEN INT
Rule 4 used :seen First Identifier

1. Valid declaration → parser accepts input.
2. Multiple identifiers → recursive rule "list → list , ID" is applied.
3. Incorrect declaration → parser reports syntax error.
